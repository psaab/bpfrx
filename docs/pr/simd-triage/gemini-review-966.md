**Verdict for #966: PLAN-KILL**

**1. Does the codebase-reality claim hold?**
Yes, but the plan document actually gives the issue author too much credit by claiming `zone_pair_policies` is a HASH map. Checking the actual code (`bpf/headers/xpf_maps.h:182-187`), it is an explicitly defined O(1) `BPF_MAP_TYPE_ARRAY` keyed by `from_zone * MAX_ZONES + to_zone`. It completely bypasses hash probing in favor of direct memory access. From there, it iterates over a rigidly bounded array capped at `MAX_RULES_PER_POLICY` (hardcoded to 256 in `bpf/headers/xpf_common.h:146`). The issue's foundational premise of a "10000 rule linear walk" is a pure hallucination disconnected from the actual architecture.

**2. Is there a real callsite where the SIMD proposal would help?**
No, the proposal is structurally impossible. The policy engine is implemented as an eBPF program (`bpf/xdp/xdp_policy.c`). The eBPF instruction set is purely scalar and does not support AVX2/AVX-512 or any SIMD vector intrinsics. You cannot compile `_mm512_cmpneq_epi32_mask` into an eBPF object. Accommodating this "refactor" would require ripping the entire policy classification engine out of kernel XDP and moving it to userspace—an apocalyptic architectural regression—just to shoehorn in vector intrinsics.

**3. Is the perf cost ≥2% on real measurement?**
Not even close. The policy evaluation isn't just below the 2% threshold; it completely fails to register on the perf radar. The entire XDP program execution accounts for only 5.94% (`bpf_prog (XDP)`), and the policy lookup slice of that is buried well below the 0.95% `mlx5e_xsk_skb_from_cqe_linear` SKB-fallback noise floor. Optimizing an unmeasurable sub-1% execution path is benchmark illiteracy.

**4. Is implementation cost worth xpf's scale?**
Absolutely not. xpf targets 22+ Gb/s with typical configurations partitioning a few dozen rules per zone pair. SIMD Bit-Vector / Tuple Space Search (like DPDK's `rte_acl`) is designed for brute-forcing flat, unpartitioned 100K+ enterprise ACLs. Absorbing massive implementation complexity, CPU-specific fallback paths, and verifier nightmares for a scale mismatch on a non-bottleneck is a catastrophic waste of engineering cycles.

**Conclusion:** 
This is an architecture-astronaut thought experiment generated by someone who didn't look at the codebase or the performance data. Kill the issue and redirect focus to the actual 13.43% `__memmove` and 9.45% RX poll bottlenecks.
