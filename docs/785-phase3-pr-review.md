# PR #796 Phase 3 MQFQ VFT — Adversarial Review

Date: 2026-04-19

## Findings

**HIGH** `queue_vtime` advances on failed pop-and-restore, so TX-ring-full retries can bias new or returning flows even though no bytes were actually drained.  
`cos_queue_pop_front()` increments `queue.queue_vtime` immediately after removing the item (`userspace-dp/src/afxdp/tx.rs:4113-4119`). Every flow-fair retry path restores the item with `cos_queue_push_front()` (`userspace-dp/src/afxdp/tx.rs:2628-2630`, `2638-2647`, `2696-2699`, `2751-2753`, `5530-5552`), but `cos_queue_push_front()` repairs only `head_finish` and `flow_bucket_bytes`; it never rewinds `queue_vtime` (`userspace-dp/src/afxdp/tx.rs:4089-4096`). Concrete case: flow A is popped during a TX-ring-full cycle and restored; `queue_vtime` becomes 1500 although zero bytes left the queue. A newly-idle flow B then enqueues at `max(0, 1500) + 1500 = 3000` via `account_cos_queue_flow_enqueue()` (`userspace-dp/src/afxdp/tx.rs:3914-3919`) instead of 1500, so B is artificially pushed behind established traffic. This also breaks the premise of the “100 Gbps x 46 years” headroom claim, because `queue_vtime` growth is no longer bounded by real drain rate.  
Mitigation: make pop-and-restore fully neutral by deferring `queue_vtime` advancement until submission succeeds, or by subtracting restored bytes on every retry path and adding a pin that asserts pre/post `queue_vtime` equality.

**LOW** The new push-front neutrality pin misses the actual retry-path defect, so CI would stay green on the bug above.  
`mqfq_push_front_is_finish_time_neutral_on_active_bucket()` captures `pre_pop_head`, `pre_pop_tail`, and `pre_pop_bytes`, then after restore checks only those fields (`userspace-dp/src/afxdp/tx.rs:10730-10755`). The test comment says the round-trip must leave the queue in “the same state” (`userspace-dp/src/afxdp/tx.rs:10688-10703`), but it never records or re-checks `queue_vtime`. On current code that omission is material: the test passes while the queue’s virtual time remains advanced by a packet that was not transmitted.  
Mitigation: extend this pin to capture `pre_pop_vtime` and assert equality after restore, and add a second variant that exercises one of the real restore helpers rather than a direct `push_front()` call.

**LOW** The “decades of headroom” pin is mostly tautological and does not bind to the production field types or the live enqueue math.  
`mqfq_finish_time_u64_has_decades_of_headroom()` computes the wrap interval from local constants and a local `u64`, not from `CoSQueueRuntime.flow_bucket_head_finish_bytes`, `flow_bucket_tail_finish_bytes`, or `account_cos_queue_flow_enqueue()` (`userspace-dp/src/afxdp/tx.rs:10764-10782`). Narrowing the production fields, changing the actual delta applied per packet, or over-advancing `queue_vtime` would not necessarily fail this test. As written it proves the arithmetic in the test body, not that the rebased implementation still has 46-year headroom.  
Mitigation: drive the real queue fields near the numeric boundary and derive the width from the actual runtime storage so a type or delta change fails immediately.

**LOW** The in-source contract is internally inconsistent after the rebase, which is risky on exactly the semantics this PR is trying to lock down.  
The `promote_cos_queue_flow_fair()` docblock still says the “current policy” is `flow_fair = queue.exact && !shared_exact` and tells reviewers to reject dropping the `!shared_exact` clause (`userspace-dp/src/afxdp/tx.rs:4924-4978`), but the implementation now sets `queue.flow_fair = queue.exact` (`userspace-dp/src/afxdp/tx.rs:4990-5006`) and the new tests require shared_exact queues to be promoted (`userspace-dp/src/afxdp/tx.rs:13217-13282`). Separately, `CoSQueueRuntime.queue_vtime` is documented as updating to the drained bucket’s finish (`userspace-dp/src/afxdp/types.rs:1088-1091`), while the code and its own new pin define it as `+= bytes_drained` (`userspace-dp/src/afxdp/tx.rs:4113-4119`, `10554-10606`). These are not runtime bugs today, but they are exactly the kind of stale invariant text that makes the next refactor regress back toward the wrong behavior.  
Mitigation: rewrite the doc comments so the source-level contract matches the rebased implementation and the new tests before calling the design settled.

## Confirmed Holds

The core byte-rate ordering logic is otherwise materially better than the old DRR path. Selection now scans `flow_bucket_head_finish_bytes` rather than ring position (`userspace-dp/src/afxdp/tx.rs:3987-3997`), enqueue splits head and tail correctly (`userspace-dp/src/afxdp/tx.rs:3894-3919`), non-drain pop advances head by the next head packet’s size (`userspace-dp/src/afxdp/tx.rs:4120-4132`), and drain-to-empty resets head and tail to zero (`userspace-dp/src/afxdp/tx.rs:3933-3946`). The equal-size, mixed-size, prepared-path, and empty-bucket pins are well targeted for those mechanisms (`userspace-dp/src/afxdp/tx.rs:10261-10334`, `10336-10437`, `10508-10551`).

The shared_exact admission downgrade did survive the rebase in both relevant code paths. `cos_queue_flow_share_limit()` returns `buffer_limit` when `queue.shared_exact` is true (`userspace-dp/src/afxdp/tx.rs:3793-3817`), and `apply_cos_admission_ecn_policy()` uses the aggregate arm whenever `flow_fair && shared_exact` (`userspace-dp/src/afxdp/tx.rs:3461-3479`). I do not see a code regression back to per-flow admission on shared_exact, although the new pin covers only the share-cap half (`userspace-dp/src/afxdp/tx.rs:10649-10685`), not the ECN branch.

The D3 rebase does not introduce any new shared mutable state into Phase 3. RSS indirection is configured in the daemon from an allowlisted interface set and either applies or restores per interface (`pkg/daemon/rss_indirection.go:111-223`); Phase 3’s new state is entirely per-queue inside `CoSQueueRuntime` (`userspace-dp/src/afxdp/types.rs:999-1105`). A concrete D3 scenario still leaves the known limitation intact: if mlx5 hashes six of twelve TCP flows onto worker 0 and one each onto workers 1-6, D3 only constrains the queue set (`pkg/daemon/rss_indirection.go:111-188`), while MQFQ orders flows only within each worker queue (`userspace-dp/src/afxdp/tx.rs:3987-3997`, `4104-4138`). That can preserve residual cross-worker CoV, but it is the pre-existing Phase 4 problem, not a new rebase regression. If the operator disables `rss-indirection`, the daemon restores the driver-default table (`pkg/daemon/rss_indirection.go:138-149`, `191-223`; `docs/785-d3-validation.md:36-58`); Phase 3 still works per worker because it does not consult any D3 global knob.

Outside MQFQ, the regression surface is narrow. The diff touches only `userspace-dp/src/afxdp/tx.rs`, `userspace-dp/src/afxdp/types.rs`, and test fixtures in `userspace-dp/src/afxdp/worker.rs`, with no production worker-path logic change in `worker.rs`. In `tx.rs`, the non-flow-fair branches still use the legacy FIFO `queue.items` path unchanged (`userspace-dp/src/afxdp/tx.rs:4001-4010`, `4024-4026`, `4044-4047`, `4101-4103`).

## Merge Readiness

NO. The rebased MQFQ head/tail ordering itself is largely coherent, and the shared_exact admission downgrade is still present, but the retry path is not actually finish-time-neutral because `queue_vtime` advances on pop and is never restored when the packet is pushed back. That is a correctness defect in the advertised TCP-pacing fairness model, and the new push-front pin misses it. I would not merge until the restore paths make `queue_vtime` neutral under TX-ring-full rollback and the tests are tightened to prove that exact invariant.
